How it behaves

1

Attention, not activity

Blocked work carries a warning border and rises to the top. Finished work fades out of the way, so the live decision stays loud.

2

One command per card

No checkbox soup. Each item offers the single next action, and taking it is a click — or a keystroke.

3

Every change leaves a receipt

Finish work, clear a blocker, add a note: each action states exactly what changed, right where you clicked. One click, or ⌘Z, puts it back.

Built to be checked

A demo that's wired to prove it holds up.

The interesting part of a portfolio piece isn't that it looks finished. It's whether it behaves, and this one is built to show its work.

Every button click-tested

CI drives a real browser that clicks every enabled control on every screen, and fails the build if one does nothing.

Two engines pinned together

Gate scripts assert the static demo and the backend produce the same workflow, on every push.

Byte-budgeted & sealed

Public files ship under size budgets, and the build fails if anything internal or oversized leaks out.

What this demo doesn't claim: real integrations, authentication, or production scale. It's sample data with no login, by design — a portfolio piece, not a product.
